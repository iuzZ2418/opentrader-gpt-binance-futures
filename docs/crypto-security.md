# 交易机器人安全与威胁模型

## 安全原则

1. GPT 是受限审批者，不是执行者；确定性代码拥有最终否决权。
2. 所有外部内容都视为不可信数据，不视为系统指令。
3. 新增风险默认关闭，减仓与紧急退出保持可用。
4. Binance、PostgreSQL 和审计日志的可核验事实优先于模型叙述。
5. paper、Demo 与 production 使用隔离的密钥、OpenAI project、数据库和主机/网络策略；
   canary/scaled/live 是 production 内仍需逐级人工解锁的资金阶段。

## 信任边界

| 边界 | 允许进入 | 明确禁止 |
|---|---|---|
| 外部来源 → 采集器 | 原始事件、来源 ID、首次观测时间 | 直接成为交易指令、执行外部仓库代码 |
| 采集器 → OpenAI | 规范化特征、授权内容、证据 ID | Binance secret、控制 token、数据库凭据 |
| OpenAI → 决策层 | 严格 schema 的审批建议 | 工具调用、任意代码、任意数量或反向开仓 |
| 决策层 → 风控 | 有效候选、决策、账户/行情快照 | 缺失 trace、过期候选、陈旧账户状态 |
| 风控 → 执行网关 | 已限制数量和退出语义的意图 | 绕过网关的 REST 请求、未授权 live 域名 |
| 控制面 → live 解锁 | 经认证的人工作业 | 单一环境变量直接开实盘 |

## 主要威胁与失败方式

| 威胁 | 预防与检测 | 失败行为 |
|---|---|---|
| X/GitHub 提示注入 | 文本作为引用数据；来源白名单、签名/ETag、内容和指令分离；schema 输出 | 拒绝新仓，保留原始来源供复核 |
| 伪造或过时来源 | 保存 source ID、URL、首次观测时间、删除/编辑事件和信誉评分 | 降权或隔离该来源，不单独产生方向 |
| 模型超时、配额或 JSON 无效 | 超时上限、严格 schema、本地二次校验 | `OPEN/ADD → REJECT`；已有仓位由硬退出管理 |
| 模型高置信错误 | 不能放大候选；0.70/0.80 门槛；ATR/敞口/回撤校验 | 缩小或拒绝，不允许亏损摊平 |
| 密钥泄漏 | 环境/密钥服务注入；IP 白名单；只开合约交易、禁提现；日志脱敏 | 撤销密钥、冻结新仓、轮换并审计 |
| 控制 API 暴露 | Compose 默认绑定 `127.0.0.1`；生产放在 mTLS/VPN/认证反代后；控制 token 定期轮换 | 阻断入口并触发 kill switch |
| 重放与时钟漂移 | Binance server-time 校准、短 `recvWindow`、唯一 client ID | 请求失败后查询状态，不增加时间窗口逃避 |
| 订单状态未知 | 用户流 + REST 双查；状态机记录 `UNKNOWN`；周期对账 | 禁止补单和同向增加风险 |
| WS 断档/乱序/重复或路由缺失 | `/public`/`market`/`private` 分流、序列号、快照重建、心跳与陈旧阈值 | 冻结新仓，保护性退出继续 |
| Algo 止损拒绝/未知归属 | 解析 `ALGO_UPDATE` 与条件触发拒绝，绑定 clientAlgoId/algoId 到追加式审计 | 熔断、REST 对账并执行风险退出 |
| 数据库/Redis 篡改或丢失 | PostgreSQL 最小权限、TLS/内部网络、备份恢复演练；消息幂等 | PostgreSQL 不可核验时停止交易 |
| 未来数据泄漏 | `event_time/observed_at` 双时间、闭合 K 线、点时交易池 | challenger 评估失败，不得晋升 |
| 风控代码被模型修改 | `StrategySpec` 参数白名单；代码部署需正常审查/测试 | 拒绝未知字段，回滚 champion |

## 凭据与网络

- Binance Demo 和 live 密钥不可复用。live 密钥限制固定出口 IP，只授予 USDⓈ-M Futures 交易权限，关闭提现。
- 机器人使用独立 Futures 账户；同一账户不得同时运行 COIN-M 机器人或保留其持仓/挂单，避免账户级持仓模式与共享限频相互干扰。
- OpenAI 的抽取、审批和研究最好使用独立 project 与预算；生产配额异常不能自动切到未批准模型。
- `.env` 只适合本地开发。生产使用 Docker secrets、云密钥服务或编排平台 secret，并保证不会出现在镜像层、命令行、提示词和错误响应中。
- PostgreSQL/Redis 不发布宿主机端口。若跨主机部署，使用私网、TLS、认证和防火墙白名单。
- API 默认仅监听宿主机回环映射。除最小 `/health` 外的交易 API 均需 token；反向代理仍须认证和限制全部入口。
- 外连防火墙仅允许批准的 Binance Futures、OpenAI 和采集源域名；Spot 端点不能被加入执行白名单。

## Live 多重门禁

生产新增风险至少需要五个条件同时成立：

1. 部署与数据库属于生产环境，且凭据为 live 专用；
2. `TRADING_STAGE=canary|scaled|live` 且资金比例严格等于 `0.10|0.25|1.0`；
3. `LIVE_TRADING_ENABLED=true`；
4. `ALLOW_BINANCE_PRODUCTION=true`；
5. 经认证的操作者调用运行时解锁，控制 token 匹配。

这五项只是必要条件。行情/账户流新鲜、对账一致、策略版本已批准、日亏损/总回撤熔断未触发、订单 trace 完整也必须通过。重启后运行时锁应恢复为关闭状态；生产密钥存在不等于允许交易。

## 隐私与外部来源

- 默认 `X_CONTENT_TO_OPENAI_ALLOWED=false`，只发送事件类型、情绪、可信度、来源 ID 和聚合特征。只有业务确认符合 X 政策与授权后才可传正文。
- 删除或编辑事件应同步到本地缓存和下游派生特征；不得使用 X 内容微调模型。
- 外部 GitHub 仓库只读抓取 Releases、Security Advisories 和批准的 commits，使用 ETag/条件请求；自有/授权仓库 webhook 还必须通过 HMAC、delivery ID、事件类型和仓库白名单；绝不安装或执行其代码。
- 搜索结果和社交热度只能缩放风险或提供证据，不能绕过量化候选独立产生方向。

## 安全验收

上线前至少回放：提示注入、伪造来源、模型超时/无效输出、密钥字段诱导、WS 乱序/重复、listen key 失效、429/418、503 unknown、部分成交、条件单拒绝、数据库断连、日亏损与 20% 回撤。每个用例必须验证“新增风险关闭、退出路径仍可用、审计链完整”。
